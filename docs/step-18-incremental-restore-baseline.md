# Step 18：可运行基线后的渐进恢复（第一轮）

## 目标

在“已回退可运行版本”的基础上，恢复核心 Agent 能力，但避免一次性恢复全部复杂交互，优先保证稳定运行与可测试。

## 本步改动

1. 补齐 Agent 结构模型
- 文件：`src/models.py`
- 新增：`AgentStep`、`ToolCallResult`、`EvaluationResult`、`AgentResult`
- 作用：让 `src/agent.py` 的输出结构可被主程序消费。

2. 补齐通用问题模式接口
- 文件：`src/ai_client.py`
- 新增：`analyze_general_question(user_question)`
- 作用：当未识别到公司时也能输出方法型解读，不再直接无结果。

3. 重建前端主入口并接入 Agent
- 文件：`app.py`
- 关键点：
  - 使用 `run_financial_agent` 作为统一入口。
  - 增加会话上下文条：显示“当前上下文公司 / 最近对比对”。
  - 增加“代理执行流程”展示。
  - 增加“延申问题建议”展示（当前为只读展示，先保稳定）。
  - 保留单公司分析的核心输出（白话解读、未来展望、评分、雷达图、日线、财务表）。

## 设计决策（为什么这样做）

1. 先恢复“可解释链路”，再恢复“自动追问交互”
- 上一阶段多次异常集中在前端复杂交互和 rerun 时序。
- 这一轮先确保：能跑、能看、能理解执行过程，再逐步加交互。

2. 重建 `app.py` 而不是继续局部补丁
- 旧文件存在编码污染，局部 patch 风险高、回归不稳定。
- 重建后结构更清晰，便于后续逐步增强。

## 验收标准

1. 应用可启动并访问 `http://localhost:8502`
2. 查询具体公司可得到分析结果
3. 顶部上下文条可反映当前公司/最近对比对
4. 未识别公司时可输出通用问题模式结果
5. 页面可看到“代理执行流程”和“延申问题建议”

## 测试指引

1. 输入：`贵州茅台`
- 预期：输出完整公司分析，并且上下文公司更新为贵州茅台。

2. 输入：`贵州茅台和宁德时代谁更好`
- 预期：出现对比结论；上下文“最近对比对”显示两家公司。

3. 输入：`怎么看现金流和利润的关系`
- 预期：进入通用问题模式，返回方法型文本，不报错。

## 风险与下一步

1. 当前“延申问题建议”为只读展示，尚未恢复一键回填与自动提交。
2. 下一步在稳定基线上恢复轻量交互（回填但不自动提交），每步都带回归测试。
