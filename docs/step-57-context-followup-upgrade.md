# Step 57 - 上下文追问能力增强（对比场景）

## 目标
在不改动现有 UI 结构的前提下，提升“已做过两公司对比后”的追问成功率：用户后续问题即使不再写公司名，也能继续沿用上文进行分析。

## 本步改动
### 1) 对比上下文自动补全
- 文件：`app.py`
- 位置：`_run_query(...)`
- 新增逻辑：
  - 当 `conversation_context.compare_pair` 已存在两家公司；
  - 且当前输入不包含 6 位代码、不包含这两家公司名字；
  - 但包含“对比/比较/谁/两家/哪个/更/vs”等对比信号；
  - 自动构造 `resolved_query = "{公司A}和{公司B}，{用户问题}"` 传入 agent。

### 2) 研究报告与主路由保持同一上下文
- 文件：`app.py`
- 变更：
  - `run_company_research_agent(...)` 改为使用 `agent_result.resolved_query or query`。
- 效果：
  - 深度研究报告与主分析共享同一“补全后问题”，避免前后结论口径不一致。

## 设计决策
- 只对“已有对比上下文 + 明确对比信号”的问题触发补全，避免误把普通通用问题强行改写。
- 仍保留原有 `run_financial_agent` 的单公司上下文补全机制，形成“双层兜底”。

## 验收标准
1. 先输入：`贵州茅台和宁德时代谁更好`
2. 再输入：`再对比一下现金流质量`
3. 预期：
   - 不需要重复公司名也能走对比分支；
   - 下方深度研究报告与主对比结论基于同一对比对象。

## 风险与后续
- 当前是规则触发，后续可升级为意图分类器驱动（减少边界误判）。
- 后续可在页面上展示“本次自动补全后的问题”，增强可解释性。
