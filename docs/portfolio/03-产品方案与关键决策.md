# 03 产品方案与关键决策

## 总体方案

采用“Agent 路由 + 工具调用 + 结构化结果对象 + 分场景渲染”的方案：
- 路由层决定走单公司、对比或通用模式。  
- 工具层负责公司识别、数据抓取、文本分析。  
- 输出层统一返回结构化 `AgentResult`。  
- 前端按场景渲染差异化页面。

## 关键决策清单

### 决策1：采用意图路由，而不是单一问答模板
- 备选方案：所有问题统一走单个回答链路。  
- 最终方案：按意图分三条路由（single/compare/general）。  
- 决策理由：不同场景输出形态差异大（例如对比场景必须双图双表），统一链路会导致体验混乱。

### 决策2：加入上下文记忆，而不是每次重新识别
- 备选方案：每轮完全独立识别，不保存状态。  
- 最终方案：保存当前公司、最近对比对、最近意图。  
- 决策理由：用户高频使用“它/这家公司”追问，缺少上下文会明显掉体验。

### 决策3：对比页采用“图表 + 表格 + 文字”组合
- 备选方案：仅输出一段结论文字。  
- 最终方案：结论卡 + 双K线 + 财务对比表 + 对比点评。  
- 决策理由：金融问题需要“证据+解释”同步出现，才能提升可信度和可读性。

### 决策4：保留失败兜底路径
- 备选方案：API 或数据失败时直接报错。  
- 最终方案：通用模式/本地兜底持续输出。  
- 决策理由：保证系统稳定可演示、可复用、可迭代。

## 取舍与影响

- 做了哪些取舍：  
  - 先做规则型 Agent，暂不做复杂多 Agent 协作。  
  - 先保证稳定可用，再迭代文本精细质量。

- 带来的收益：  
  - 交付速度快，功能闭环清晰。  
  - 对比链路成功率高（当前基线 100%）。

- 遗留问题：  
  - 单公司意图识别还有提升空间。  
  - 通用问答文本质量稳定性仍需优化。
