# Step 1 说明文档：建立 Agent 化基线与文档日志体系

## 1. 目标

在不大改业务能力的前提下，把当前项目从“可运行的财报解读器”升级为“可工程化推进的 Agent 基线版本”。

本步聚焦三个交付物：

- 最小 Agent 编排入口：`run_financial_agent(query)`
- 前端可见的 Agent 执行过程（trace）
- 双轨记录体系：阶段说明文档 + 当日工作日志

## 2. 当前现状（Step 1 前）

- 查询链路：`app.py` 直接调用数据层和 AI 层，缺少统一编排入口。
- 展示链路：页面只显示分析结果，不显示“系统如何执行”。
- 记录链路：有日常开发日志，但缺少“单步骤可交接说明文档”。

## 3. SOP 拆解（Step 1 版本）

本步将查询流程固化为 3 段最小 SOP：

1. 意图识别：根据用户问题识别为财报解读/风险分析/对比/通用问答。
2. 公司识别与数据抓取：调用现有数据层生成 `snapshot`。
3. 财报解读生成：调用现有 AI 层返回 `analysis`。

设计理由：

- 先可见、后复杂。优先把执行过程透明化，便于演示和排障。
- 不替换原有模块，降低回归风险，保证迭代速度。

## 4. 设计决策

- 新增 `src/agent.py` 作为最小编排层，统一入口 `run_financial_agent(query)`。
- 扩展 `src/models.py`，增加 `AgentStep` 与 `AgentResult`，供编排层和前端复用。
- 修改 `app.py` 查询入口，先执行 Agent，再复用现有结果渲染。
- 新增“离线演示模式”提示：未走真实 API 时明确提示当前为兜底解读。

不做项（明确边界）：

- 不引入重型 Agent 框架
- 不修改 OpenAI 调用协议
- 不改动现有免责声明逻辑
- 不扩展多轮对话和复杂工具链

## 5. 本步变更清单

- 新增最小 Agent 编排模块：`src/agent.py`
- 扩展数据结构：`src/models.py`
- 前端接入 Agent trace 展示：`app.py`
- 新增本说明文档：`docs/step-01-agent-baseline.md`
- 新增当日日志：`dev_logs/2026-05-20.md`

## 6. 验收标准

满足以下条件可判定 Step 1 完成：

- 应用可启动，`http://localhost:8502` 可访问。
- 输入已识别公司时可完成分析并展示原有结果。
- 页面可见最小 Agent 执行过程，至少包含 3 步：意图识别、取数、解读。
- 无 API Key 场景下仍可完成流程并展示离线演示提示。

## 7. 测试指引

建议测试输入：

- `贵州茅台`
- `帮我看看000001`
- 未配置 API Key 时再测一次上述任一输入

预期：

- 页面顶部先出现 Agent 执行过程卡片
- 结果区保持原有白话解读、评分、图表和风险提示
- 离线场景出现“离线演示模式”提示

## 8. 风险与下一步

已知风险：

- 当前意图识别是规则法，覆盖率有限，复杂问题识别精度一般。
- Agent trace 为最小实现，尚未包含工具级输入输出详情。

Step 2 建议：

- 补齐工具注册层（ToolRegistry）和更细粒度的 step trace
- 增加输出质量评估（合规、字段完整性、可解释性）
- 将 README 的端口说明与实际启动端口统一
